# ☕🔮 Coffee Oracle Bot

**Telegram-бот для гадания на кофейной гуще с использованием AI Vision**

Бот анализирует фотографии кофейной гущи через OpenAI Vision API и генерирует мистические позитивные предсказания в стихах. Включает полноценную админ-панель с аналитикой, управлением пользователями и настройками.

Приложение состоит из трёх основных компонентов, запускаемых параллельно через `ApplicationOrchestrator`:

│  │  (aiogram)   │  │  (FastAPI +  │  │  (SQLite + │ │

│  └──────┬───────┘  └──────┬───────┘  └─────┬─────┘ │
│         │                 │                 │       │

│              ┌───────▼───────┐              │       │

### Поток обработки предсказания

```
Пользователь отправляет фото
        │
        ▼
MediaGroupMiddleware (сбор группы фото, 1с таймаут)
        │
        ▼
PhotoProcessor (скачивание, ресайз, сохранение на диск)
        │
        ▼

        │

        │
        ▼
Markdown → Telegram HTML конвертация
        │
        ▼

```

## Установка и запуск

### Предварительные требования

- Python 3.11+
- Telegram Bot Token (от @BotFather)
- OpenAI API Key с доступом к моделям Vision

### Локальный запуск

```bash
# 1. Клонирование
git clone <repository-url>
cd coffee-oracle

# 2. Создание виртуального окружения
python -m venv venv
source venv/bin/activate  # Linux/macOS
# venv\Scripts\activate   # Windows

# 3. Установка зависимостей
pip install -r requirements.txt

# 4. Конфигурация
cp .env.example .env
# Отредактируйте .env — заполните обязательные переменные

# 5. Запуск
python -m coffee_oracle.main
```

### Docker

```bash
# Сборка
docker build -t coffee-oracle .

# Запуск
docker run -d \
  --name coffee-oracle \
  --env-file .env \
  -p 8000:8000 \
  -v ./data:/app/data \
  -v ./media:/app/media \
  -v ./logs:/app/logs \
  coffee-oracle
```

### Логика обработки фото

1. Пользователь отправляет одно фото или группу фото (media group)
2. `MediaGroupMiddleware` собирает все фото группы (таймаут 1 секунда)
3. Бот показывает индикатор набора текста (typing)
4. `PhotoProcessor` скачивает фото через Telegram API, ресайзит и сохраняет на диск
5. `LLMClient` отправляет base64-изображение(я) в OpenAI Vision API
6. Ответ проходит фильтрацию негативного контента
7. Markdown конвертируется в Telegram HTML
8. Предсказание отправляется пользователю (с разбиением, если >4096 символов)
9. Предсказание сохраняется в базу данных

## Админ-панель

Доступна по адресу `http://<host>:<ADMIN_PORT>/`. Защищена HTTP Basic Auth.

### Страницы

| URL | Описание |
|-----|----------|
| `/` | Дашборд — KPI-карточки, графики активности, таблица ретенции |
| `/users` | Список пользователей с поиском и пагинацией |
| `/predictions` | Список предсказаний с просмотром полного текста и фото |
| `/settings` | Редактор настроек бота (системный промпт, температура и др.) |
| `/health` | Health-check эндпоинт |

### Дашборд

Отображает четыре KPI-метрики: общее количество пользователей, новые пользователи за 24 часа, общее количество предсказаний и предсказания за 24 часа. Включает интерактивный график активности с переключением периодов (24 часа, 7 дней, 4 недели, 12 месяцев) и таблицу ретенции. Данные автоматически обновляются каждые 5 минут.

### Настройки

Через админ-панель можно изменять без перезапуска бота: системный промпт, температуру генерации, приветственное сообщение, инструкцию для фото, сообщение обработки, текст "О боте", режим анализа нескольких фото и фильтр негативного контента. После сохранения кэш настроек автоматически очищается.

## Обработка фотографий

### PhotoProcessor

Класс `PhotoProcessor` отвечает за полный цикл работы с фото.

**Скачивание:** фото загружается через Telegram Bot API (`bot.get_file` → `bot.download_file`).

**Ресайз:** изображения приводятся к максимальному размеру 800×800 пикселей с сохранением пропорций. Формат всегда конвертируется в JPEG. Прозрачные PNG автоматически получают белый фон. Если файл превышает 4 МБ после ресайза, качество JPEG итеративно снижается с шагом 10 (от 85 до 50).

**Сохранение:** файлы сохраняются в `/app/media/` с UUID-именами (`{uuid4}.jpg`).

**Валидация:** фото отклоняются, если размер превышает 20 МБ. Из массива `PhotoSize` выбирается самое большое (по `file_size`).

### Обработка нескольких фото

При отправке media group все фото обрабатываются параллельно. Настройка `analyze_all_photos` определяет, отправлять ли все изображения в один запрос к API или только первое.

## Интеграция с OpenAI

### LLMClient

Класс `LLMClient` инкапсулирует работу с OpenAI API.

**Формат запроса:** системный промпт из настроек + пользовательское сообщение с base64-изображениями. В системный промпт автоматически инжектируются текущая дата, время, день недели и время суток (по московскому часовому поясу), а также имя пользователя.

**Фильтрация:** ответ проверяется на наличие негативных слов. Если обнаружены слова из стоп-списка (болезнь, смерть, неудача и др.), возвращается безопасное fallback-предсказание.

**Кэширование настроек:** настройки из базы данных кэшируются в глобальном словаре `_settings_cache`. Кэш очищается через API или при сохранении настроек в админ-панели.

**Обработка ошибок:** rate limit, authentication errors, bad request — каждый тип ошибки выбрасывает `OpenAIError` с пользовательски-дружелюбным сообщением.

## Middleware

### MediaGroupMiddleware

Telegram отправляет каждое фото из media group как отдельное сообщение. `MediaGroupMiddleware` решает эту проблему.

**Механизм:** при получении первого фото с `media_group_id` создаётся запись в глобальном словаре и запускается отложенная задача с таймаутом 1 секунда. Последующие фото с тем же `media_group_id` добавляются в ту же запись. По истечении таймаута все собранные фото передаются в обработчик одним пакетом.

**Данные в контексте обработчика:**

- `media_group_photos` — список всех сообщений с фото
- `is_media_group` — `True`, если фото пришли группой
- `media_group_caption` — подпись из любого фото группы

**Очистка:** после обработки запись удаляется из словаря. Поздно прибывшие фото (после `processed=True`) игнорируются.

## Обработка ошибок

### Иерархия исключений

```
CoffeeOracleError (базовый)
├── DatabaseError
├── OpenAIError
├── PhotoProcessingError
├── ConfigurationError
└── AuthenticationError
```

Каждое исключение содержит два поля: `message` (пользовательски-дружелюбный текст) и `details` (технические детали для логов). Функция `format_error_message()` выбирает нужный формат в зависимости от контекста.

В обработчике фото ошибки `PhotoProcessingError` и `OpenAIError` перехватываются и отображаются пользователю. Остальные ошибки ловятся общим `except Exception` с дженерик-сообщением.

## Утилиты

### telegram.py

- `markdown_to_telegram_html(text)` — конвертирует Markdown в подмножество HTML, поддерживаемое Telegram (`<b>`, `<i>`, `<s>`, `<code>`, `<pre>`, `<a>`). Сначала экранируются спецсимволы HTML, затем применяются regex-замены. Результат проходит через `sanitize_telegram_html()` для закрытия незакрытых тегов.
- `split_message(text, max_length=4096)` — разбивает длинное сообщение на чанки с учётом лимита Telegram. Пытается разбить по параграфам, затем по строкам, затем по словам.
- `strip_html_tags(text)` — fallback для случаев, когда Telegram не может распарсить HTML. Удаляет все теги и восстанавливает HTML-entities.
- `sanitize_telegram_html(text)` — исправляет незакрытые и несовпадающие HTML-теги, пропускает только разрешённые теги Telegram.

### logging.py

Настройка логирования с ротацией файлов: максимум 10 МБ на файл, 1 бэкап. Логи пишутся в stdout и в файл. Логгеры aiogram, httpx и uvicorn.access приглушены до уровня WARNING.

## Деплой

### Персистентные данные

При деплое необходимо смонтировать три директории:

- `data/` — файл SQLite базы данных
- `media/` — загруженные фотографии пользователей
- `logs/` — файлы логов

### Порты

- `8000` (или `ADMIN_PORT`) — админ-панель и API
- Telegram-бот работает через Long Polling и не требует открытых портов

### Сигналы завершения

Приложение корректно обрабатывает `SIGTERM` и `SIGINT`: останавливает бота, закрывает сессию Telegram, останавливает Uvicorn-сервер и закрывает соединение с базой данных.

---

## Ограничения

- SQLite не подходит для высоконагруженных сценариев. Для масштабирования замените на PostgreSQL (потребуется изменить `database_url` и драйвер на `asyncpg`)
- Фото хранятся на локальном диске. Для масштабирования рассмотрите S3-совместимое хранилище
- Автоочистка фото срабатывает при превышении лимита в 20 000 записей
